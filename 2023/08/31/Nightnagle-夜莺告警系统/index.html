Nightingale 夜莺与告警系统

前言

夜莺监控是一款开源云原生观测分析工具，采用 All-in-One 的设计理念，集数据采集、可视化、监控告警、数据分析于一体，与云原生生态紧密集成，提供开箱即用的企业级监控分析和告警能力。它支持管理多个时序库，支持可视化的灵活告警策略同时适配多个集群，支持监控的能力开放给公司所有研发团队，让研发团队自助服务。在这些场景下，夜莺监控是个不错的选择。

选择夜莺监控的原因

通常来说，聊到监控系统绕不过的就是zabbix和prometheus。
Zabbix 是过去二十年里开源监控系统的代表作，有着广泛的用户基础，但由于架构在当前云原生时代的格格不入显得尤为辛苦，当你选择在容器架构下，把众多微服务的监控数据往 Zabbix 数据模型上套，会发现非常辛苦，甚至于很多时候根本都行不通。
以公司目前的生产环境为例。目前的zabbix是实体机部署，在主机数量和为1000台左右，监控器数量为200000左右，触发器数量7000上下的条件下由于关系型数据库的瓶颈限制已经出现多次查询超时导致数据无法刷新的情况。

而prometheus在监控领域的已经成为事实上监控系统的代表。prometheus之后的任何一个告监控系统都兼容prometheus的数据模型。
prometheus由于使用的tsdb时序数据库，本身更适合监控数据的扩展。从测试结果来看，在硬件资源满足的情况下。prometheus 单实例在每秒采集 10万条监控数据的情况下，在数据处理和查询方面依然有着不错的性能表现；而在数据的长期存储方面，thanos和victoriametrics可以很好的解决这个问题。而在当前生产环境中，由于存在多大10余套的promethus like 实例。这也导致了使用和管理成本高的问题。
不过prometheus在公司当前生产环境的运行中，目前存在以下痛点：

  • 现有开源组合方案 Prometheus + Alertmanager + Grafana 多个系统较为割裂，无法开箱即用;
  • 通过修改配置文件来管理 Prometheus、Alertmanager ，学习曲线大，用户量多时，协同有难度;
  • 数据量过大而无法有效扩展 Prometheus 集群；
  • 生产环境运行多套 Prometheus 集群，面临分区管理和使用成本高的问题；

而开源夜莺监控系统正好可以解决这里的大多数问题

  1. 夜莺监控系统支持接入promtheus like数据源，提供统一，灵活的数据查询能力。
  2. 夜莺监控系统支持自定义告警通知和全局告警转发，可以同时兼容目前公司的告警网关
  3. 夜莺监控系统支持个人定制化告警，可以监控的能力开放给公司所有研发团队，让研发团队自助配置告警规则及查看历史告警，支持多种告警推送渠道，企业微信应用，企业微信机器人，邮箱，短信，电话。

下面我们可以详细的看看夜莺监控系统和我们在实际生产环境对它做的告警改造集成。

系统架构


首先上图中间的飞鸟代表夜莺的核心进程 n9e （下文以 n9e 代替），它的集群方式非常简单只需部署多节点即可实现。

对于 n9e 来说，它本身依赖的存储有两个

  • Mysql : 存放配置类别信息，如用户，监控大盘，告警规则等
  • Redis : 存放访问令牌(JWT Token)，心跳信息，如机器列表中CPU、内存、时间偏移、核数、操作系统、CPU架构等

而对于上面数据流图来说

  • Nightingale 可以接收各类 agent 作为采集器上报的监控数据，比如 datadog-agent、grafana-agent、telegraf、categraf
  • Nightingale 往 mysql（也可以使用pg）的 target 表存储机器列表和心跳信息，并且通过这个信息判断机器是 UP 还是 DOWN
  • Nightingale 把机器列表中操作系统、CPU 架构信息、CPU 利用率、内存利用率、时间偏移等元数据存储在 Redis 里
  • Nightingale 把接收到的监控数据（时序数据）转发给后端时序库，时序库可以并行写入多个，比如 VictoriaMetrics、Prometheus、M3DB、Thanos 等

可以看到夜莺的整体架构并不复杂，夜莺作为核心的告警引擎使用配置的规则实时进行监控查询。当然在有多个边缘机房和中心机房网络链路不是很好的情况下，需要添加n9e-edge(类似prometheus的 pushgateway)作为数据中转来做边缘下沉式混杂部署方案

在边缘机房在和中心机房网络连接比较好的情况下，目标机器只需部署 Categraf，直连中心机房即可（公网一定要注意添加好安全相关配置）。

可是当边缘机房和中心机房网络链路不是很好的情况下，除了目标机器部署 Categraf 外，还推荐把告警引擎（n9e-alert）和时序数据库一起下沉部署。n9e-alert 是 n9e 的一个只保留告警引擎模块的独立可执行程序

当前公司生产环境部署的话，n9e 前面架设负载均衡（harpoxy或nginx都可以）。让 agent 通过负载均衡上报监控数据，上层访问 n9e 也通过负载均衡，这样 n9e 任何一个实例挂掉，不会影响到整个服务可用性，从而做到高可用。而且集群中多个 n9e 实例会均分告警规则，分片处理，从而可以处理更大量的告警规则。

部署问题汇总

1.Nightanagle夜莺监控系统V5及V6版本间的差异 –v5版本api，server模块在v6合并同一模块，官方暂无文档参考，接入生产环境数据源，配置高可用告警引擎。
2、官方提供的数据库初始化语句问题，MySQL8.0.20执行初始化SQL报错定位报错语句：insert into busi_group_member(busi_group_id, user_group_id, perm_flag) values(1, 1, “rw”);
报错信息：Unknown column ‘rw’ in ‘field list’
原因分析：”rw”使用了双引号
3、监控系统Bug问题导致异常崩溃 – 查询源码定位故障代码位置，调试后解决，提issue
例如
监控系统有时告警会出现“invalid memory address or nil pointer的日志报错，经定位是某个未初始化的变量，向官方提交pull request 后修复

5、部署VictoriaMetrics集群采集categraf客户端信息 –考虑VictoriaMetrics存储3个月历史数据，缓解prometheus压力

总的来说，在部署过程中遇到的问题并不算多，大多是监控系统版本差异及少数代码bug的解决。

夜莺监控系统的简单使用

经过上述的介绍及改造流程，终于到了使用阶段，这里主要给大家介绍一下最重要也是最常用告警使用部分
目前告警有两种方案：
一种兼容当前运维部使用的告警中心告警。即每个部门的告警中心群。一般情况下服务器的通用告警会发送到这个群
还有一种是通过夜莺监控系统自身的告警发送。支持自定义告警通知方式，合作方私有云告警采用此方案，也支持个人及项目组使用。

总结

对于当前的夜莺监控系统，已经实现了在当前生产环境的稳定运行，统一汇总了各个prometheus的查询入口，基本代替了alertmanager，有了可视化的告警页面，实现了与当前告警网关的集成。提供个人定制化告警，将告警能力开放给所有开发同事，有效提高了运维效率。
